# 巡检宝页面原型结构

## 1. 信息架构

```text
巡检宝
├── 首页驾驶舱
├── 小区管理
│   ├── 小区列表
│   └── 小区详情
├── 道路街面
│   ├── 道路列表
│   └── 道路详情
├── 重点点位
│   ├── 点位列表
│   └── 点位详情
├── 巡检报告
│   ├── 报告列表
│   └── 报告详情
├── 问题台账
│   ├── 问题列表
│   └── 问题详情
├── 文件资料库
└── 系统管理
    ├── 用户管理
    └── 基础字典
```

## 2. 全局布局

### 2.1 桌面端后台

建议采用左侧导航 + 顶部工具栏 + 主内容区。各一级模块采用独立页面跳转，不做成一个长页面上下滑。

```text
┌────────────────────────────────────────────────────────────┐
│ 顶部栏：巡检宝 / 曲阳路街道 / 搜索 / 用户                  │
├──────────────┬─────────────────────────────────────────────┤
│ 左侧导航      │ 主内容区                                     │
│ 首页          │ index.html                                  │
│ 小区管理      │ communities.html                            │
│ 道路街面      │ roads.html                                  │
│ 巡检报告      │ reports.html                                │
│ 问题台账      │ issues.html                                 │
│ 地图资产      │ map-assets.html                             │
└──────────────┴─────────────────────────────────────────────┘
```

设计建议：

- 色彩以白色、浅灰、深蓝灰为主，状态色使用红、橙、绿、蓝。
- 页面不要做夸张科技风，保持政府工作平台的稳重感。
- 表格和筛选项要清晰，方便高频查询。
- 图表不要太多，首页保留真正有管理价值的指标。
- 一级导航点击后进入独立页面，避免所有内容堆在首页里上下滚动。

### 2.2 手机端 / 小程序

底部导航建议 4 个入口：

```text
首页 / 小区 / 报告 / 问题
```

手机端重点是查阅，不做复杂管理。

## 3. 首页驾驶舱原型

### 3.1 页面目标

让街道管理人员在 10 秒内看懂近期巡检情况。

### 3.2 页面结构

```text
┌────────────────────────────────────────────────────────────┐
│ 曲阳路街道无人机巡检概览                      日期筛选      │
├────────────────────────────────────────────────────────────┤
│ [本月巡检] [发现问题] [已整改] [待处理] [复查通过]           │
├────────────────────────────┬───────────────────────────────┤
│ 街道二维地图                 │ 问题类型分布                    │
│ - 小区名称标签               │ - 飞线                         │
│ - 道路名称标签               │ - 违建                         │
│ - 重点问题点位               │ - 占道经营                     │
│                              │ - 广告牌                       │
├────────────────────────────┼───────────────────────────────┤
│ 高频问题小区                 │ 最新巡检报告                    │
│ 1. XX 小区                   │ 2026-xx-xx 小区巡检报告         │
│ 2. XX 小区                   │ 2026-xx-xx 道路巡检报告         │
├────────────────────────────┴───────────────────────────────┤
│ 重点待处理问题列表                                           │
└────────────────────────────────────────────────────────────┘
```

### 3.3 首页组件

- 首屏核心指标：本月巡检、发现问题、待处理，放在主视觉区域内。
- 地图组件：加载街道二维底图，按小区、道路、点位显示名称、范围和问题。
- 分类图表：问题类型占比。
- 排行榜：高发小区、高发道路。
- 最新报告：最近上传的报告。
- 重点问题：严重、超期、反复出现的问题。

### 3.4 交互

- 点击指标卡进入对应筛选结果。
- 点击小区名称或小区范围进入小区详情。
- 点击道路名称或道路热区进入道路详情。
- 点击地图问题点位打开问题简要信息。
- 点击小区排行进入小区详情。
- 点击报告打开 PDF 预览。

### 3.5 二维地图加载原型

```text
┌────────────────────────────────────────────────────────────┐
│ 地图工具栏：小区图层 道路图层 点位图层 问题状态 搜索         │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  街道总览底图                                               │
│                                                            │
│  ┌─────────────┐       ═════════ 曲阳路 ═════════           │
│  │  XX 小区     │  ● 待处理问题                              │
│  └─────────────┘                                            │
│                                                            │
│          ┌─────────────┐                                    │
│          │  XX 花园     │                                    │
│          └─────────────┘                                    │
│                                                            │
├────────────────────────────────────────────────────────────┤
│ 悬浮信息：对象名称 / 本月问题 / 待处理 / 最新巡检             │
└────────────────────────────────────────────────────────────┘
```

地图支持：

- 上传街道总览图。
- 上传单个小区地图。
- TIF 上传后后台转换为网页展示图。
- 在地图上绘制小区边界、道路热区和点位。
- 每个热区绑定一个小区、道路或点位档案。
- 名称标签常驻显示，鼠标悬浮显示简要统计。
- 点击后跳转对应详情页。

## 11. 关键设计建议

### 11.1 按对象组织，不按文件夹组织

客户脑子里想的是“这个小区怎么样”“这条路怎么样”，不是“某个文件在哪个文件夹里”。所以平台应该用小区、道路、点位做入口，文件只是附属资料。

### 11.2 首页不要堆所有数据

首页只放对管理决策有用的内容：

- 发现了多少问题。
- 哪些问题还没处理。
- 哪些小区或道路高发。
- 最近有什么报告。
- 地图上问题集中在哪里。

首页不再单独放一排“覆盖小区、覆盖道路、高频反复点位、复查通过”巡检指标卡。这类内容后续可以作为统计分析页或详情页里的辅助信息，不占用首页主流程。

### 11.3 小程序只保留高频查看

手机端不适合复杂编辑。第一版小程序建议只做查看、搜索、预览和下载。

### 11.4 报告和问题必须互相关联

每份报告要能看到关联问题，每个问题也要能回到关联报告。这样客户查资料时不会断。

### 11.5 为后续闭环预留状态字段

即使第一版不做派单，也要先保留问题状态、整改照片、复查记录等字段，后续升级成本会低很多。

### 11.6 地图用作入口，不只做背景图

首页地图应该成为客户进入数据的主要入口之一。小区、道路、重点点位都要有可点击热区，地图上的名称标签要和后台档案绑定。这样客户看到地图时，可以直接点“小区名”进入小区巡检资料，点“道路名”进入道路街面资料，点“问题点”进入问题详情。

### 11.7 TIF 作为原始资产，网页展示用转换结果

TIF 适合保存高清原始地图，但不适合作为客户浏览时直接加载的图片。第一版建议后台保留原始 TIF，同时生成 PNG / WebP 预览图；如果地图很大，再生成瓦片。前端只加载预览图或瓦片，保证打开速度和手机端体验。
