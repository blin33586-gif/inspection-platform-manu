# 巡检宝开发落地规格

## 1. 第一版系统边界

第一版目标是把巡检宝从静态原型推进到可开发的业务系统蓝图，优先完成“资料沉淀、客户查看、问题台账、地图入口”四件事。

第一版包含：

- 管理后台。
- 客户查看端。
- 文件上传与预览。
- 小区、道路、点位档案。
- 巡检报告管理。
- 问题台账管理。
- 首页地图入口。
- 基础统计。

第一版不包含：

- AI 自动识别。
- 派单流转。
- 自动生成执法文书。
- 无人机实时直播。
- 多街道复杂组织架构。
- 三维地图。

## 2. 推荐工程结构

后续进入真实开发时，建议从静态原型升级为前后端分离结构。

```text
inspection-platform-manu
├── apps
│   ├── admin-web          # 管理后台和客户 Web 查看端
│   └── miniapp            # 微信小程序查看端，第二阶段进入
├── services
│   └── api                # 后端 API 服务
├── packages
│   ├── shared             # 共享类型、枚举、工具方法
│   └── ui                 # 通用 UI 组件，后续再抽
├── docs                   # 产品、设计、接口和阶段记录
├── prototype              # 当前静态原型，作为视觉和结构参考
└── infra                  # 部署、数据库、对象存储配置
```

第一版如果追求速度，也可以先做：

```text
apps/admin-web
services/api
docs
prototype
```

## 3. 技术选型建议

### 3.1 前端

建议：

- React。
- TypeScript。
- Ant Design。
- Vite。
- ECharts。
- 地图交互第一版使用自定义图片地图 + SVG/Canvas 热区。

原因：

- 后台管理组件成熟。
- 表格、筛选、上传、弹窗、标签页开发快。
- 适合政务类管理系统。
- 后续接地图和统计容易。

### 3.2 后端

建议：

- Node.js。
- NestJS。
- Prisma 或 TypeORM。
- PostgreSQL。

原因：

- 模块化清晰。
- 适合做 REST API。
- TypeScript 前后端类型一致。
- 后续接对象存储、任务队列、权限都方便。

### 3.3 文件存储

建议：

- 开发环境：本地文件存储。
- 生产环境：阿里云 OSS、腾讯云 COS 或华为云 OBS。

原则：

- 数据库只保存文件元数据和访问地址。
- 原始大文件不放数据库。
- TIF 原始文件保留，网页展示用转换后的 WebP/PNG/瓦片。

### 3.4 部署

第一版部署建议：

- 云服务器：1 台应用服务器。
- 数据库：云数据库 PostgreSQL 或 MySQL。
- 文件：对象存储。
- 域名：HTTPS。
- 备份：数据库每日备份，文件对象存储版本管理。

## 4. 核心业务对象

### 4.1 管理对象

管理对象分为：

- 小区。
- 道路。
- 重点点位。

这三类对象是平台的主要入口。报告、问题、图片、地图热区都应绑定到这些对象上。

### 4.2 巡检成果

巡检成果分为：

- 巡检记录。
- 巡检报告。
- 原始照片。
- 标注照片。
- 视频片段。
- 地图成果。

### 4.3 问题闭环

问题闭环第一版只做基础状态：

- 待处理。
- 处理中。
- 已整改。
- 复查通过。
- 暂不处理。
- 已归档。

第二阶段再做派单、责任部门、整改期限、超期提醒。

## 7. 数据权限

第一版角色：

- 平台管理员。
- 项目运营。
- 街道客户。
- 城管查看。

权限规则：

- 平台管理员：所有功能。
- 项目运营：增删改查、上传文件、维护地图。
- 街道客户：查看、筛选、下载。
- 城管查看：查看问题、报告和照片。

第一版先不做复杂到部门和居委的细粒度权限，但数据结构预留 `tenant_id` 和 `street_id`，方便后续复制到其他街道。

## 8. 文件上传流程

### 8.1 普通附件

流程：

1. 前端选择文件。
2. 后端生成上传凭证或接收文件。
3. 文件保存到本地或对象存储。
4. 后端写入附件表。
5. 业务对象绑定附件。

支持类型：

- PDF。
- JPG。
- PNG。
- WebP。
- MP4。
- TIF / TIFF。

### 8.2 报告上传

报告上传时应同时支持：

- PDF 文件。
- 封面图。
- 报告摘要。
- 关联对象。
- 关联问题。
- 关联照片。

第一版可以先上传 PDF 和摘要，关联问题后续在报告详情里补。

### 8.3 照片上传

照片需要保存：

- 原图。
- 缩略图。
- 拍摄时间。
- 关联对象。
- 关联问题。
- 是否为整改前。
- 是否为整改后。

## 9. TIF 地图处理流程

### 9.1 第一版处理方式

第一版不直接在浏览器加载 TIF。

流程：

1. 后台上传 TIF 原始文件。
2. 服务端保存原始 TIF。
3. 服务端生成 WebP 或 PNG 预览图。
4. 前端加载预览图。
5. 后台在预览图上绘制小区、道路、点位热区。
6. 首页地图读取热区并支持点击跳转。

### 9.2 大图处理方式

如果 TIF 文件较大：

1. 保留原始 TIF。
2. 生成低清预览图。
3. 生成地图瓦片。
4. 前端按缩放级别加载瓦片。

### 9.3 GeoTIFF 处理方式

如果 TIF 是 GeoTIFF：

- 保存地理坐标信息。
- 生成瓦片。
- 支持经纬度点位叠加。
- 后续可接高德或标准 GIS 图层。

第一版先不强制要求 GeoTIFF。

## 10. 地图热区规则

热区类型：

- 点：重点点位、问题点。
- 线：道路。
- 多边形：小区范围。
- 矩形：临时框选。

热区必须绑定：

- 对象类型。
- 对象 ID。
- 显示名称。
- 显示颜色。
- 是否显示。

点击规则：

- 小区热区跳转小区详情。
- 道路热区跳转道路详情。
- 点位热区跳转重点点位详情。
- 问题点跳转问题详情。

## 11. API 风格

建议使用 REST API，统一前缀：

```text
/api/v1
```

统一返回格式：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

分页格式：

```json
{
  "items": [],
  "page": 1,
  "pageSize": 20,
  "total": 0
}
```

错误码建议：

- `0`：成功。
- `40001`：参数错误。
- `40101`：未登录。
- `40301`：无权限。
- `40401`：资源不存在。
- `50001`：系统错误。

## 13. 验收标准

### 13.1 首页

- 能看到本月巡检、发现问题、待处理。
- 能看到街道地图。
- 能点击小区进入小区档案。
- 能点击道路进入道路档案。
- 能看到问题类型分布。

### 13.2 小区档案

- 能查看小区列表。
- 能查看小区详情。
- 能查看该小区报告。
- 能查看该小区问题。
- 能查看该小区照片。

### 13.3 道路街面

- 能查看道路列表。
- 能查看道路详情。
- 能维护广告牌和占道点位。
- 能查看道路问题。

### 13.4 巡检报告

- 能上传 PDF 报告。
- 能预览报告。
- 能下载报告。
- 能绑定小区、道路或点位。

### 13.5 问题台账

- 能新增问题。
- 能筛选问题。
- 能查看问题详情。
- 能更新问题状态。
- 能上传问题照片。

### 13.6 地图资产

- 能上传地图文件。
- 能生成网页预览图。
- 能维护热区。
- 首页能读取热区。

## 14. 开发顺序

建议按以下顺序进入真实开发：

1. 建前端工程和后端工程。
2. 建数据库和基础枚举。
3. 做用户登录。
4. 做小区、道路、点位基础数据。
5. 做报告和附件上传。
6. 做问题台账。
7. 做首页统计。
8. 做地图资产和热区。
9. 做小程序查看端。
10. 做部署和备份。

## 15. 阶段冻结规则

每个阶段完成后：

- 本地测试通过。
- 写入开发阶段记录。
- Git 提交。
- 推送到 GitHub。
- 后续模块只引用已完成结果，不反复重做已定型模块。
