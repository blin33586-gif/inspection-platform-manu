# 巡检宝数据库与接口草案

## 1. 数据库设计原则

第一版数据库要支持一个街道样板，但字段上预留多街道、多客户复制能力。

核心原则：

- 所有业务表保留 `id`、`created_at`、`updated_at`。
- 需要逻辑删除的表保留 `deleted_at`。
- 可复制到多街道的表保留 `tenant_id`、`street_id`。
- 文件只保存元数据和 URL，不保存文件二进制。
- 问题、报告、附件都通过对象类型和对象 ID 绑定小区、道路、点位。

## 6. 首页统计口径

### 6.1 本月巡检

统计当前街道当月 `inspection_records` 数量。

### 6.2 发现问题

统计当前街道当月 `issues.found_at` 落在当月的问题数量。

### 6.3 待处理

统计当前街道 `issues.status` 为 `pending` 或 `processing` 的数量。

### 6.4 高频对象

按 `issues.object_type + issues.object_id` 聚合，取问题数量前 5。

### 6.5 问题分布

按 `issues.category` 聚合。

## 7. 索引建议

建议优先建立：

```text
issues(street_id, found_at)
issues(object_type, object_id)
issues(status)
issues(category)
reports(street_id, report_date)
reports(related_object_type, related_object_id)
attachments(business_type, business_id)
map_hot_areas(map_asset_id)
```
